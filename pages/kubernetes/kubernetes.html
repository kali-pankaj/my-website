📦 What is Kubernetes?

Kubernetes (K8s) is an open-source container orchestration platform used to manage containerized applications.

It automates deployment, scaling, and management of containers across a cluster of machines.

Key Idea

Containers are usually created using tools like Docker, and Kubernetes is used to manage those containers efficiently.

Why Kubernetes is Used

Kubernetes helps run applications across multiple machines while ensuring:

  • ✔ High Availability
  • ✔ Scalability
  • ✔ Reliability

Kubernetes Handles Tasks Like:

  • Deploying and managing containers (e.g., Docker)
  • Scaling applications up or down automatically
  • Load balancing network traffic
  • Managing networking and storage
  • Maintaining the desired state of applications

Kubernetes Architecture

⚙️ Kubernetes Architecture Components

The Kubernetes architecture consists of two main parts:

  • 1. Control Plane (Master Node)
  • 2. Worker Nodes

1️⃣ Control Plane (Master Node)

The Control Plane manages the entire Kubernetes cluster. It makes global decisions such as scheduling pods, maintaining the desired state, and managing cluster operations.

🔹 API Server (kube-apiserver)

  • The API Server is the main entry point for the Kubernetes cluster.
  • It exposes the Kubernetes API used by users, CLI tools, and components.
  • All communication between components passes through the API Server.
  • It processes REST requests and updates cluster state.

🔹 etcd

  • etcd is a distributed key-value store.
  • It stores all cluster data such as configuration and metadata.
  • Information about pods, deployments, services, and nodes is stored here.
  • It is critical for maintaining the cluster’s desired state.

🔹 Controller Manager (kube-controller-manager)

  • The controller manager ensures the cluster stays in the desired state.
  • It continuously watches the cluster and fixes differences automatically.

Important Controllers:

  • ReplicaSet Controller – Maintains the required number of pod replicas.
  • Deployment Controller – Manages deployment updates.
  • Job Controller – Handles batch jobs.
  • Node Controller – Monitors node health.

🔹 Scheduler (kube-scheduler)

  • The scheduler assigns pods to worker nodes.
  • It selects the best node based on resources such as CPU, memory, and policies.
  • It ensures efficient workload distribution.

🔹 Cloud Controller Manager

  • Handles cloud-provider specific operations.
  • Manages resources like load balancers, storage volumes, and networking.
  • Used when Kubernetes runs on cloud platforms such as AWS, GCP, or Azure.

2️⃣ Worker Nodes

Worker nodes are where the actual applications (containers) run. Each worker node contains components that allow Kubernetes to run workloads.

🔹 Kubelet

  • The kubelet is an agent running on every worker node.
  • It ensures containers are running inside pods.
  • It communicates with the API Server to receive instructions.

🔹 Kube Proxy

  • kube-proxy manages networking rules.
  • It enables communication between pods and services.
  • It routes traffic using iptables or IPVS.

🔹 Container Runtime

  • The container runtime is responsible for running containers.
  • It pulls images, creates containers, and manages their lifecycle.
  • Common runtimes include containerd and CRI-O.

🔹 Pods

  • Pods are the smallest deployable unit in Kubernetes.
  • A pod can contain one or more containers.
  • Containers in the same pod share network and storage.
  • Each pod receives its own IP address.

📚 Other Key Kubernetes Concepts

🔹 Namespaces

Namespaces are used to logically divide a Kubernetes cluster into multiple virtual clusters. This allows different teams, projects, or environments to share the same physical Kubernetes cluster while keeping their resources isolated.

Each namespace contains its own set of resources such as Pods, Services, Deployments, and ConfigMaps. This makes it easier to organize workloads and manage permissions.

Namespaces help in:

  • Organizing resources – Resources can be grouped based on project or team.
  • Isolation – Applications in one namespace do not interfere with others.
  • Access control – Permissions can be applied per namespace using RBAC.
  • Resource management – Resource quotas can limit CPU and memory usage.
  • Multi-tenant environments – Multiple teams can safely use the same cluster.

Default Namespaces in Kubernetes

  • default – The namespace used when no namespace is specified while creating resources.
  • kube-system – Contains important Kubernetes system components like DNS, scheduler, and controller manager.
  • kube-public – A namespace that is readable by all users. Used for cluster information that needs to be publicly accessible.
  • kube-node-lease – Used by Kubernetes nodes to send heartbeat signals to the control plane for node health monitoring.

In short: Namespaces organize and isolate resources inside a Kubernetes cluster.

🔹 Services

A Service in Kubernetes is an abstraction that defines a logical group of Pods and provides a stable network endpoint to access them.

Pods are dynamic and may be created, destroyed, or replaced at any time. Because of this, their IP addresses constantly change. A Kubernetes Service solves this problem by providing a permanent IP address and DNS name.

A Service provides:

  • Stable IP Address – Even if pods restart.
  • DNS Name – Services can be accessed using service-name.namespace.
  • Load Balancing – Traffic is distributed across multiple pods.
  • Service Discovery – Applications can find each other easily.

Services use labels and selectors to identify which pods should receive traffic. Whenever new pods are created or removed, the service automatically updates its routing configuration.

In short: Services provide stable communication between pods and applications.

Service Types in Kubernetes

1️⃣ ClusterIP (Default)
  • This is the default service type.
  • The service is only accessible from inside the cluster.
  • Other pods communicate using the internal cluster network.
  • Commonly used for internal microservices communication.
2️⃣ NodePort
  • Exposes the service on a specific port of every worker node.
  • The port range is 30000 – 32767.
  • Users can access the service using the node's IP address.
  • Mostly used for development or testing environments.
3️⃣ LoadBalancer
  • Creates an external load balancer provided by the cloud platform.
  • Automatically distributes traffic across pods.
  • Commonly used for production web applications.
4️⃣ ExternalName
  • Maps a Kubernetes service to an external DNS name.
  • Allows applications inside the cluster to access external services.
  • Example: connecting to an external database.
5️⃣ Headless Service
  • Created by setting ClusterIP: None.
  • Does not assign a single service IP.
  • DNS returns the IP addresses of all pods.
  • Used in StatefulSets like Kafka, Cassandra, or databases.

🔹 Deployments

A Deployment is a higher-level Kubernetes object that manages the lifecycle of pods and ReplicaSets.

Deployments are commonly used for stateless applications and provide advanced features for managing updates and scaling.

Main features of Deployments:

  • Scaling – Increase or decrease the number of running pods.
  • Rolling Updates – Update application versions without downtime.
  • Rollback – Revert to a previous version if an update fails.
  • Self-healing – Automatically replace failed pods.

🔹 ReplicaSets

A ReplicaSet ensures that a specified number of pod replicas are running at any given time.

If a pod crashes or is deleted, the ReplicaSet automatically creates a new pod to maintain the desired number of replicas.

ReplicaSets are usually managed indirectly through Deployments rather than created manually.

🔹 StatefulSets

A StatefulSet is used for stateful applications that require stable identities and persistent storage.

Unlike Deployments, StatefulSets maintain unique network identities and persistent volumes for each pod.

Common use cases:

  • Databases (MySQL, PostgreSQL)
  • Distributed systems (Kafka, Cassandra)
  • Applications requiring stable storage

🔹 DaemonSets

A DaemonSet ensures that a specific pod runs on all nodes (or selected nodes) in the cluster.

Whenever a new node joins the cluster, the DaemonSet automatically schedules a pod on that node.

Common use cases:

  • Log collection agents (Fluentd)
  • Monitoring agents (Prometheus Node Exporter)
  • Network plugins

🔹 Jobs and CronJobs

Kubernetes Jobs are used for running tasks that must complete successfully.

  • Job – Runs a task until it finishes successfully. Example: database migration or batch processing.
  • CronJob – Runs jobs on a schedule similar to Linux cron. Example: daily backup tasks.

🔹 Ingress

An Ingress manages external access to services in the cluster. It allows HTTP and HTTPS traffic to reach internal services.

Ingress also provides advanced routing capabilities and centralized control over incoming traffic.

Features:

  • HTTP/HTTPS routing
  • SSL termination
  • Load balancing
  • Host-based routing
  • Path-based routing

🌐 Kubernetes Networking

Pod-to-Pod Networking

In Kubernetes, every pod receives a unique IP address. Pods can communicate with each other directly without NAT.

Service Discovery

Kubernetes provides built-in DNS that automatically assigns DNS names to services.

Pods can communicate using service names instead of IP addresses, making applications easier to manage.

Network Policies

Network Policies define rules that control communication between pods.

They allow administrators to specify which pods can communicate with each other based on labels, namespaces, and ports.

📍 Affinity and AntiAffinity

Node Affinity

Node Affinity allows pods to be scheduled on specific nodes based on node labels.

  • Example: Run pods only on nodes with SSD storage.
  • Helps control where workloads run.

Pod Affinity

Pod Affinity schedules pods close to other pods.

  • Used when applications need low latency communication.
  • Example: Application pods running close to cache pods.

Pod AntiAffinity

Pod AntiAffinity ensures pods are scheduled away from certain pods.

  • Improves high availability.
  • Prevents all replicas from running on the same node.

Pod Affinity vs Node Affinity

Type Meaning Based On
Node Affinity Run pods on specific nodes Node Labels
Pod Affinity Run pods near specific pods Pod Labels
Pod AntiAffinity Avoid running near specific pods Pod Labels

🚀 Amazon EKS Cluster Upgrade Guide

Upgrading an Amazon EKS cluster is an important operational task that ensures your Kubernetes environment stays secure, stable, and compatible with the latest Kubernetes features. Each Kubernetes release introduces security patches, performance improvements, and sometimes breaking API changes. Because of this, cluster upgrades must be performed carefully and in a controlled sequence.

A typical EKS upgrade process involves the following stages:

  • Preparing the cluster for upgrade
  • Upgrading cluster add-ons
  • Upgrading the Kubernetes control plane
  • Upgrading worker nodes
  • Validating workloads after the upgrade

1️⃣ Pre-Upgrade Checklist

Before upgrading your EKS cluster, it is critical to complete several preparation steps. These checks ensure that the upgrade does not introduce unexpected failures or downtime for running workloads.

🔹 Backup Cluster State

In a self-managed Kubernetes cluster, the etcd database stores the entire cluster state. This includes information such as:

  • Cluster configuration
  • Pod definitions
  • Secrets and ConfigMaps
  • Service configurations
  • Node information

If the cluster is self-managed (for example, installed using kubeadm or running on-premises), administrators must create a backup of the etcd database before performing an upgrade. This backup allows the cluster to be restored if an upgrade fails or introduces configuration issues.

However, in Amazon EKS the Kubernetes control plane and etcd database are fully managed by AWS. This means administrators do not directly manage etcd backups. AWS automatically ensures high availability and reliability of the control plane.

🔹 Check Current Kubernetes Version

Before performing an upgrade, administrators must determine the current Kubernetes version running in the cluster. Kubernetes upgrades must follow a sequential minor version path. This means you cannot skip minor versions during upgrades.

For example:

  • Kubernetes v1.31 → v1.32 (Allowed)
  • Kubernetes v1.31 → v1.33 (Not allowed directly)

You can check the cluster version using the following command: 

eksctl get cluster --name <cluster-name> --region <region>

This command displays important information about the cluster including:

  • Cluster name
  • Kubernetes version
  • Cluster region
  • Cluster status

You can also verify the Kubernetes version using the AWS Management Console:

  • Open the AWS Console
  • Navigate to EKS
  • Select your cluster
  • View the Kubernetes version in the cluster details

🔹 Review Official Kubernetes and EKS Documentation

Before upgrading to a new Kubernetes version, administrators should always review the official release notes and documentation. Each Kubernetes version may introduce changes that affect cluster behavior or application compatibility.

Release notes typically include:

  • New features added in the version
  • Deprecated APIs
  • Removed functionality
  • Security improvements
  • Performance optimizations

Reviewing these changes helps ensure that your workloads, CI/CD pipelines, and infrastructure tools remain compatible with the new version.

🔹 Document Running Add-ons

Most Kubernetes clusters rely on several add-ons that extend core cluster functionality. These components handle networking, DNS, monitoring, autoscaling, and service communication.

Before upgrading the cluster, it is important to identify all running add-ons and confirm that they support the target Kubernetes version.

Use the following command to list system components:

kubectl get deployments -n kube-system

Key add-ons to verify:

  • CoreDNS
    Provides DNS resolution inside the Kubernetes cluster so that services can communicate using service names instead of IP addresses.
  • VPC CNI (Amazon VPC CNI Plugin)
    Handles pod networking by assigning IP addresses from the AWS VPC network to Kubernetes pods.
  • Metrics Server
    Collects resource usage metrics such as CPU and memory. These metrics are required for autoscaling components.
  • Horizontal Pod Autoscaler (HPA)
    Automatically increases or decreases the number of pods based on resource usage metrics.
  • Istio
    A service mesh used for traffic management, security policies, and observability between microservices.
  • KEDA
    Kubernetes Event Driven Autoscaling, used for event-based scaling such as message queues.
  • Karpenter
    A modern node autoscaler that dynamically provisions worker nodes based on workload demands.

2️⃣ Upgrade Add-ons First

Add-ons should be upgraded before upgrading the Kubernetes control plane. This ensures that networking, DNS, and autoscaling components remain compatible with the newer Kubernetes version.

Amazon EKS supports managed add-ons that can be upgraded easily using the eksctl CLI.

Upgrade Managed Add-ons Using eksctl

eksctl update addon --name coredns \
--cluster <cluster-name> \
--region <region> \
--force

eksctl update addon --name kube-proxy \
--cluster <cluster-name> \
--region <region> \
--force

eksctl update addon --name vpc-cni \
--cluster <cluster-name> \
--region <region> \
--force

Upgrading these add-ons ensures that cluster networking, service routing, and DNS resolution continue working correctly after the control plane upgrade.

Manually Upgrade Other Add-ons

Some add-ons are not managed by EKS and must be upgraded manually. Examples include Istio, Prometheus, KEDA, and other monitoring tools.

Administrators should follow the official upgrade documentation for each component to ensure compatibility with the new Kubernetes version. 

kubectl apply -f https://github.com/aws/amazon-vpc-cni-k8s/releases/download/v1.11.7/amazon-vpc-cni-k8s-1.11.7.yaml

3️⃣ Upgrade the EKS Control Plane

After all add-ons have been upgraded, the next step is upgrading the Kubernetes control plane. The control plane includes components such as:

  • Kubernetes API Server
  • Controller Manager
  • Scheduler
  • etcd database

In Amazon EKS, the control plane is fully managed by AWS. When you upgrade the control plane, AWS automatically upgrades these components while maintaining cluster availability.

Upgrade Using eksctl

eksctl upgrade cluster --name <cluster-name> --region <region>

This upgrade process is non-disruptive, meaning that running workloads continue operating during the upgrade.

Alternative: Upgrade via AWS Console

  • Open AWS Console
  • Navigate to EKS → Clusters
  • Select your cluster
  • Click "Update Version"
  • Select the target Kubernetes version

EKS upgrades must follow sequential versions and cannot skip minor versions.

4️⃣ Upgrade Worker Nodes

After upgrading the control plane, worker nodes must also be upgraded so that their kubelet version matches the control plane version.

Create a New Node Group

eksctl create nodegroup \
--cluster <cluster-name> \
--name <new-nodegroup-name> \
--node-type t3.medium \
--nodes 3 \
--nodes-min 1 \
--nodes-max 5 \
--node-ami-family AmazonLinux2 \
--version <target-k8s-version> \
--region <region>

Cordon and Drain Old Nodes

kubectl cordon <old-node-name>

kubectl drain <old-node-name> \
--ignore-daemonsets \
--delete-emptydir-data

Delete Old Node Group

eksctl delete nodegroup \
--cluster <cluster-name> \
--name <old-nodegroup-name>

Self-Managed Node Groups:

For self-managed node groups (EC2 instances in an Auto Scaling group):

  • 1.Update the Launch Template with the latest EKS-optimized AMI for the new Kubernetes version.
  • 2. Update the Auto Scaling Group to use the updated AMI.
  • 3. Cordon and Drain Old Nodes as with managed node groups.

5️⃣ Post-Upgrade Validation

After completing the upgrade, administrators must validate that the cluster and workloads are functioning correctly.

Check Node Status

kubectl get nodes

Check Pod Status

kubectl get pods --all-namespaces

Verify System Add-ons

kubectl get deployment -n kube-system

If all nodes are in the Ready state and pods are running without errors, the upgrade is considered successful.

🎯 Interview Question

Interviewer: Can you walk me through the process of upgrading a Kubernetes cluster on EKS?

Answer:

  • Pre-Upgrade: First, I check the current Kubernetes version using eksctl, review release notes, and identify all running add-ons such as CoreDNS, VPC CNI, Metrics Server, Istio, KEDA, and Karpenter.
  • Upgrade Add-ons: Next, I upgrade critical managed add-ons using eksctl to ensure compatibility with the new Kubernetes version.
  • Upgrade Control Plane: Then I upgrade the control plane using eksctl or the AWS Console. This step is non-disruptive, so workloads keep running.
  • Upgrade Worker Nodes: After that, I create a new node group with the updated version and migrate workloads by draining and deleting the old nodes.
  • Post-Upgrade Validation: Finally, I verify node status, pod health, and add-on functionality to ensure the cluster is operating correctly.
  • Rollback Strategy: If issues occur, I can revert add-on versions or recreate node groups because EKS does not support direct rollback of cluster versions.
