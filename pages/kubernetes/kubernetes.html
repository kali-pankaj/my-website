📦 Kubernetes Overview

📦 What is Kubernetes?

Kubernetes (K8s) is an open-source container orchestration platform used to manage containerized applications.

It automates deployment, scaling, and management of containers across a cluster of machines.

💡 Key Idea

Containers are created using tools like Docker, while Kubernetes manages those containers efficiently across systems.

🎯 Why Kubernetes is Used

Kubernetes ensures applications run reliably across multiple machines:

  • ✔ High Availability
  • ✔ Scalability
  • ✔ Reliability

⚙️ What Kubernetes Handles

  • Deploying and managing containers (Docker)
  • Auto scaling applications
  • Load balancing traffic
  • Managing networking & storage
  • Maintaining desired state

⚙️ Kubernetes Architecture Components

The Kubernetes architecture consists of two main parts:

  • 1. Control Plane (Master Node)
  • 2. Worker Nodes

🧠 Control Plane

The Control Plane manages the cluster and makes global decisions like scheduling and maintaining the desired state.

🖥 Worker Nodes

Worker nodes run actual applications (containers) and execute workloads.

⚙️ Kubernetes Architecture

Kubernetes architecture consists of two main components: Control Plane (Master Node) and Worker Nodes.

1️⃣ Control Plane (Master Node)

The Control Plane is responsible for managing the entire cluster. It handles scheduling, maintains the desired state, and manages cluster operations.

🔹 API Server

  • Main entry point of cluster
  • Handles REST API requests
  • All communication passes through it
  • Updates cluster state

🔹 etcd

  • Distributed key-value store
  • Stores cluster data
  • Pods, services, nodes info
  • Maintains desired state

🔹 Controller Manager

  • Maintains desired state
  • Watches cluster continuously
  • Fixes differences automatically

Controllers:

  • ReplicaSet Controller
  • Deployment Controller
  • Job Controller
  • Node Controller

🔹 Scheduler

  • Assigns pods to nodes
  • Uses CPU, memory, policies
  • Ensures efficient distribution

🔹 Cloud Controller

  • Cloud-specific operations
  • Manages LB, storage, networking
  • Used in AWS, GCP, Azure

2️⃣ Worker Nodes

Worker nodes are where applications actually run. They contain components that execute workloads and maintain communication with the control plane.

🔹 Kubelet

  • Node agent
  • Ensures containers are running
  • Communicates with API Server

🔹 Kube Proxy

  • Manages networking rules
  • Handles pod-to-service communication
  • Uses iptables/IPVS

🔹 Container Runtime

  • Runs containers
  • Pulls images
  • Manages lifecycle
  • Examples: containerd, CRI-O

🔹 Pods

  • Smallest deployable unit
  • Can have multiple containers
  • Shared network & storage
  • Each pod has unique IP

📚 Namespaces

Namespaces divide a Kubernetes cluster into logical groups for better organization, isolation, and resource management.

Namespaces allow multiple teams or environments to share the same cluster while keeping their resources isolated.

📦 Organization

Group resources by project/team

🔒 Isolation

Separate workloads securely

🔑 Access Control

RBAC per namespace

📊 Resource Control

Limit CPU & memory usage

👥 Multi-Tenant

Multiple teams share cluster

🔹 Default Namespaces

default

Used when no namespace is specified

kube-system

System components (DNS, scheduler)

kube-public

Publicly readable resources

kube-node-lease

Node heartbeat tracking

Namespaces provide logical separation and help manage large Kubernetes environments efficiently.

🌐 Kubernetes Services

A Service in Kubernetes provides a stable network endpoint to access a dynamic set of Pods. It enables reliable communication inside and outside the cluster.

A Service is an abstraction that defines a logical group of Pods and exposes them using a stable IP address and DNS name.

Since Pods are ephemeral (they can be created, deleted, or replaced), their IP addresses keep changing. Services solve this problem by providing a permanent endpoint.

🔹 What a Service Provides

📌 Stable IP

Remains constant even if pods restart

🌍 DNS Name

Access using service-name.namespace

⚖ Load Balancing

Distributes traffic across pods

🔎 Service Discovery

Pods can find each other easily

👉 Services use labels & selectors to dynamically map traffic to Pods.

🔹 How Services Work

Client Request
➡
Service (Stable IP)
➡
Pod (via Labels)

Traffic is automatically routed to healthy pods using label selectors.

🔹 Service Types

1️⃣ ClusterIP (Default)

  • Accessible only inside cluster
  • Used for internal communication
  • Default service type

2️⃣ NodePort

  • Exposes service on node IP
  • Port range: 30000–32767
  • Used for testing/dev

3️⃣ LoadBalancer

  • Creates cloud load balancer
  • Public access to application
  • Used in production

4️⃣ ExternalName

  • Maps to external DNS
  • No proxying inside cluster
  • Used for external services

5️⃣ Headless Service

  • ClusterIP: None
  • Returns all pod IPs
  • Used with StatefulSets

📌 Summary

Services provide stable networking, load balancing, and service discovery in Kubernetes, ensuring reliable communication between applications.

📍 Affinity and Anti-Affinity

Kubernetes Affinity rules control how pods are scheduled across nodes. They help optimize performance, reliability, and availability by defining placement preferences.

🔹 Node Affinity

Node Affinity allows pods to be scheduled on specific nodes based on node labels.

  • Run pods only on selected nodes
  • Based on node labels (e.g., SSD, region)
  • Improves workload placement control
👉 Example: Schedule pods only on nodes with SSD storage

🔹 Pod Affinity

Pod Affinity schedules pods close to other pods for better communication.

  • Used for low-latency communication
  • Groups related services together
  • Improves performance
👉 Example: App pods placed near Redis cache pods

🔹 Pod Anti-Affinity

Pod Anti-Affinity ensures pods are scheduled away from certain pods.

  • Prevents same pods from running on same node
  • Improves high availability
  • Avoids single point of failure
👉 Example: Spread replicas across different nodes

🔹 Affinity Comparison

Type Meaning Based On
Node Affinity Run pods on specific nodes Node Labels
Pod Affinity Run pods near specific pods Pod Labels
Pod Anti-Affinity Avoid running near certain pods Pod Labels
👉 Node Affinity = Node based | Pod Affinity = Pod based

🚀 Amazon EKS Cluster Upgrade Guide

Upgrading an Amazon EKS cluster is a critical operational task that ensures your Kubernetes environment remains secure, stable, and compatible with the latest features.

Each Kubernetes release introduces:

🔐 Security patches
⚡ Performance improvements
🆕 New features
⚠️ API changes
👉 Upgrades must be done carefully to avoid downtime or failures.

🔹 Upgrade Workflow

1️⃣ Pre-Upgrade Preparation
2️⃣ Upgrade Add-ons
3️⃣ Upgrade Control Plane
4️⃣ Upgrade Worker Nodes
5️⃣ Post-Upgrade Validation
👉 Always follow this sequence to ensure a safe upgrade process.

1️⃣ Pre-Upgrade Checklist

Upgrading an EKS cluster is a critical operation that must be carefully planned to avoid service disruption. A proper pre-upgrade checklist ensures cluster stability, workload compatibility, and minimal downtime during the upgrade process.

🔹 Backup Cluster State

In a self-managed Kubernetes environment, the etcd database acts as the central source of truth for the cluster. It stores all critical state data required for cluster recovery and operation.

  • Cluster configuration and API objects
  • Pod specifications and deployments
  • Sensitive data such as Secrets and ConfigMaps
  • Service discovery and networking configurations
  • Node metadata and cluster topology

Before performing any upgrade in a self-managed cluster (kubeadm/on-prem), administrators must take a snapshot backup of etcd. This ensures that the cluster can be restored in case of upgrade failures, misconfigurations, or data corruption.

👉 In Amazon EKS, the control plane (including etcd) is fully managed. AWS automatically handles backups, replication, and high availability, so manual intervention is not required.

🔹 Check Current Kubernetes Version

Kubernetes enforces a sequential version upgrade policy. This means clusters must be upgraded one minor version at a time to maintain compatibility between the control plane and worker nodes.

✅ Allowed Upgrade Path

v1.31 → v1.32

❌ Unsupported Upgrade

v1.31 → v1.33 (Skipping versions is not allowed)

To verify the current cluster version and metadata, use the following command: 

eksctl get cluster --name <cluster-name> --region <region>
  • Cluster name and status
  • Current Kubernetes version
  • AWS region and availability zones
  • Cluster health and endpoint details
👉 Alternatively, verify via AWS Console → EKS → Cluster → Details tab.

🔹 Review Official Documentation

Before upgrading, always review the official Kubernetes and EKS release notes for the target version. This helps identify breaking changes and compatibility issues.

✨ Newly introduced features and enhancements
⚠️ Deprecated APIs that may be removed soon
❌ Removed or unsupported functionalities
🔐 Security patches and vulnerability fixes
⚡ Performance improvements and optimizations

Understanding these changes ensures that your applications, CI/CD pipelines, and infrastructure configurations continue to work seamlessly after the upgrade.

🔹 Document Running Add-ons

Kubernetes clusters rely heavily on add-ons for networking, DNS resolution, monitoring, and autoscaling. Before upgrading, all add-ons must be identified and verified for compatibility with the target Kubernetes version. 

kubectl get deployments -n kube-system
CoreDNS

Handles internal DNS resolution for service-to-service communication.

VPC CNI

Manages pod networking by assigning IP addresses from the VPC.

Metrics Server

Collects CPU and memory usage metrics required for autoscaling.

HPA (Horizontal Pod Autoscaler)

Automatically scales pods based on resource utilization.

Istio

Provides service mesh capabilities like traffic routing and observability.

KEDA

Enables event-driven autoscaling based on external triggers.

Karpenter

Dynamically provisions nodes based on workload demand.

👉 Always upgrade add-ons before upgrading the control plane to avoid compatibility issues.

2️⃣ Upgrade Add-ons First

Before upgrading the Kubernetes control plane, it is essential to upgrade all cluster add-ons. These components handle networking, DNS, and service communication, and must remain compatible with the target Kubernetes version.

📘 Best Practice: Always upgrade add-ons before the control plane to avoid compatibility issues.

🔹 Why Upgrade Add-ons First?

Add-ons are tightly coupled with Kubernetes APIs. If they are not upgraded, they may break due to deprecated or removed APIs in newer Kubernetes versions.

🌐 Networking (VPC CNI)
📡 DNS Resolution (CoreDNS)
🔀 Service Routing (kube-proxy)
📊 Monitoring & Autoscaling
👉 Failure to upgrade add-ons can lead to pod networking issues, DNS failures, or service downtime.

🔹 Upgrade Managed Add-ons (eksctl)

Amazon EKS provides managed add-ons that can be upgraded easily using the eksctl CLI. 

eksctl update addon --name coredns \
--cluster <cluster-name> \
--region <region> \
--force

eksctl update addon --name kube-proxy \
--cluster <cluster-name> \
--region <region> \
--force

eksctl update addon --name vpc-cni \
--cluster <cluster-name> \
--region <region> \
--force
CoreDNS

Handles DNS resolution inside the cluster

kube-proxy

Manages service networking and routing

VPC CNI

Provides pod networking using AWS VPC

👉 Use --force to overwrite existing versions if required.

🔹 Manually Upgrade Other Add-ons

Some add-ons are not managed by EKS and must be upgraded manually using their official installation manifests or Helm charts.

Istio

Service mesh for traffic management

Prometheus

Monitoring and alerting system

KEDA

Event-driven autoscaling

Follow official documentation for each tool to ensure compatibility with the Kubernetes version you are upgrading to. 

kubectl apply -f https://github.com/aws/amazon-vpc-cni-k8s/releases/download/v1.11.7/amazon-vpc-cni-k8s-1.11.7.yaml
👉 Always verify version compatibility before applying updates.

3️⃣ Upgrade the EKS Control Plane

After upgrading all add-ons, the next step is upgrading the Kubernetes control plane. This ensures that core Kubernetes components are updated to the target version while maintaining cluster availability.

📘 Official Docs: https://docs.aws.amazon.com/eks/latest/userguide/update-cluster.html

🔹 Control Plane Components

The Kubernetes control plane consists of critical components responsible for cluster management and orchestration:

  • Kubernetes API Server
  • Controller Manager
  • Scheduler
  • etcd database
👉 In Amazon EKS, these components are fully managed by AWS.

🔹 Upgrade Using eksctl

You can upgrade the control plane using the eksctl CLI. 

eksctl upgrade cluster \
--name <cluster-name> \
--region <region>
👉 This process is non-disruptive, meaning workloads continue running.

🔹 Upgrade via AWS Console

  • Open AWS Console
  • Go to EKS → Clusters
  • Select your cluster
  • Click Update Version
  • Select target Kubernetes version
👉 Upgrades must follow sequential minor versions (no skipping).

4️⃣ Upgrade Worker Nodes

After upgrading the control plane, worker nodes must be upgraded to match the Kubernetes version to ensure compatibility and stability.

🔹 Create New Node Group

eksctl create nodegroup \
--cluster <cluster-name> \
--name <new-nodegroup-name> \
--node-type t3.medium \
--nodes 3 \
--nodes-min 1 \
--nodes-max 5 \
--node-ami-family AmazonLinux2 \
--version <target-k8s-version> \
--region <region>
👉 Always create a new node group instead of upgrading in-place.

🔹 Cordon and Drain Old Nodes

kubectl cordon <old-node-name>

kubectl drain <old-node-name> \
--ignore-daemonsets \
--delete-emptydir-data
👉 This safely moves workloads to new nodes.

🔹 Delete Old Node Group

eksctl delete nodegroup \
--cluster <cluster-name> \
--name <old-nodegroup-name>

🔹 Self-Managed Node Groups

  • Update Launch Template with latest EKS AMI
  • Update Auto Scaling Group
  • Cordon and drain old nodes

5️⃣ Post-Upgrade Validation

After completing the upgrade, validate that all cluster components and workloads are functioning correctly.

🔹 Verification Commands

kubectl get nodes
kubectl get pods --all-namespaces
kubectl get deployment -n kube-system
👉 All nodes should be in Ready state and pods running.

🎯 Interview Question

Q: Can you explain EKS upgrade process?

  • Pre-Upgrade: Check version, review docs, list add-ons
  • Add-ons: Upgrade CoreDNS, kube-proxy, VPC CNI
  • Control Plane: Upgrade using eksctl/console
  • Nodes: Create new node group, drain old nodes
  • Validation: Verify nodes and pods
  • Rollback: Recreate nodes / downgrade add-ons
👉 EKS does NOT support direct control plane rollback.
