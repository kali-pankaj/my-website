📦 Kubernetes Overview

Learn the fundamentals of Kubernetes, its purpose, and architecture.

📦 What is Kubernetes?

Kubernetes (K8s) is an open-source container orchestration platform used to manage containerized applications.

It automates deployment, scaling, and management of containers across multiple machines.

💡 Key Idea

Containers are created using tools like Docker, while Kubernetes manages them efficiently.

🎯 Why Use Kubernetes?

Kubernetes ensures applications run reliably:

  • High Availability
  • Scalability
  • Reliability

⚙️ What Kubernetes Handles

  • Container deployment and management
  • Auto-scaling applications
  • Load balancing
  • Networking and storage
  • Maintaining desired state

⚙️ Architecture Overview

Kubernetes architecture has two main components:

  • Control Plane (Master Node)
  • Worker Nodes

🧠 Control Plane

Manages the cluster, schedules workloads, and maintains the desired state.

🖥 Worker Nodes

Run the actual applications (containers) and execute workloads.

⚙️ Kubernetes Architecture

Kubernetes architecture consists of two main components: Control Plane (Master Node) and Worker Nodes.

1️⃣ Control Plane (Master Node)

The Control Plane manages the entire cluster. It handles scheduling, maintains the desired state, and manages cluster operations.

🔹 API Server

  • Main entry point of cluster
  • Handles REST API requests
  • All communication passes through it
  • Updates cluster state

🔹 etcd

  • Distributed key-value store
  • Stores cluster data
  • Pods, services, nodes info
  • Maintains desired state

🔹 Controller Manager

  • Maintains desired state
  • Watches cluster continuously
  • Fixes differences automatically

Controllers:

  • ReplicaSet Controller
  • Deployment Controller
  • Job Controller
  • Node Controller

🔹 Scheduler

  • Assigns pods to nodes
  • Uses CPU, memory, policies
  • Ensures efficient distribution

🔹 Cloud Controller

  • Cloud-specific operations
  • Manages load balancer, storage, networking
  • Used in AWS, GCP, Azure

2️⃣ Worker Nodes

Worker nodes run applications and handle workloads. They communicate with the control plane.

🔹 Kubelet

  • Node agent
  • Ensures containers are running
  • Communicates with API Server

🔹 Kube Proxy

  • Manages networking rules
  • Handles pod-to-service communication
  • Uses iptables/IPVS

🔹 Container Runtime

  • Runs containers
  • Pulls images
  • Manages lifecycle
  • Examples: containerd, CRI-O

🔹 Pods

  • Smallest deployable unit
  • Can have multiple containers
  • Shared network & storage
  • Each pod has unique IP

📚 Namespaces

Namespaces divide a Kubernetes cluster into logical groups for better organization, isolation, and resource management.

Namespaces allow multiple teams or environments to share the same cluster while keeping their resources isolated.

  • 📦 Organization - Group resources by project/team
  • 🔒 Isolation - Separate workloads securely
  • 🔑 Access Control - RBAC per namespace
  • 📊 Resource Control - Limit CPU & memory usage
  • 👥 Multi-Tenant - Multiple teams share cluster

🔹 Default Namespaces

  • default - Used when no namespace is specified
  • kube-system - System components (DNS, scheduler)
  • kube-public - Publicly readable resources
  • kube-node-lease - Node heartbeat tracking

👉 Namespaces provide logical separation and help manage large Kubernetes environments efficiently.

🌐 Kubernetes Services

A Service in Kubernetes provides a stable network endpoint to access Pods. It ensures reliable communication inside and outside the cluster.

📌 What is a Service?

A Service is an abstraction that groups Pods and exposes them using a stable IP address and DNS name.

Pods are temporary and their IPs change, but Services provide a fixed endpoint.

🔹 What a Service Provides

  • 📌 Stable IP - Remains constant
  • 🌍 DNS Name - service-name.namespace
  • ⚖ Load Balancing - Distributes traffic
  • 🔎 Service Discovery - Easy communication

👉 Services use labels and selectors to route traffic to Pods.

🔹 How Service Works

Client → Service (Stable IP) → Pod (via labels)

🔹 Service Types

1️⃣ ClusterIP (Default)

  • Only inside cluster
  • Used for internal communication

2️⃣ NodePort

  • Exposes service on node IP
  • Port range: 30000–32767

3️⃣ LoadBalancer

  • Creates cloud load balancer
  • Public access

4️⃣ ExternalName

  • Maps to external DNS
  • No internal proxy

5️⃣ Headless Service

  • ClusterIP: None
  • Returns Pod IPs directly

📌 Summary

Services provide stable networking, load balancing, and service discovery in Kubernetes.

📍 Affinity and Anti-Affinity

Affinity rules control how Pods are scheduled across nodes. They help improve performance, reliability, and availability.

🔹 Node Affinity

Node Affinity allows Pods to run on specific nodes based on labels.

  • Run pods on selected nodes only
  • Based on node labels (SSD, region)
  • Improves workload placement

👉 Example: Run pods only on SSD nodes

🔹 Pod Affinity

Pod Affinity places Pods close to other related Pods.

  • Used for low-latency communication
  • Keeps related services together
  • Improves performance

👉 Example: App pods near Redis pods

🔹 Pod Anti-Affinity

Pod Anti-Affinity ensures Pods are placed away from each other.

  • Prevents same pods on same node
  • Improves high availability
  • Avoids single point of failure

👉 Example: Spread replicas across nodes

🔹 Comparison

Type Meaning Based On
Node Affinity Run pods on specific nodes Node Labels
Pod Affinity Run pods near other pods Pod Labels
Pod Anti-Affinity Avoid certain pods Pod Labels

👉 Node Affinity = Node based | Pod Affinity = Pod based

🚀 Amazon EKS Cluster Upgrade Guide

Upgrading an Amazon EKS cluster ensures your Kubernetes environment is secure, stable, and up-to-date with the latest features.

Each Kubernetes release includes:

  • 🔐 Security patches
  • ⚡ Performance improvements
  • 🆕 New features
  • ⚠️ API changes

👉 Upgrades must be done carefully to avoid downtime or failures.

🔹 Upgrade Workflow

  1. Pre-Upgrade Preparation
  2. Upgrade Add-ons
  3. Upgrade Control Plane
  4. Upgrade Worker Nodes
  5. Post-Upgrade Validation

👉 Always follow this sequence for a safe upgrade.

1️⃣ Pre-Upgrade Checklist

Upgrading an EKS cluster must be carefully planned to avoid downtime and ensure compatibility.

🔹 Backup Cluster State

In self-managed Kubernetes, etcd is the cluster's database (source of truth).

  • Cluster configuration and API objects
  • Pods and deployments
  • Secrets and ConfigMaps
  • Networking and services
  • Node information

👉 Always take an etcd backup before upgrading (for kubeadm/on-prem clusters).

👉 In EKS, AWS manages etcd automatically (no manual backup needed).

🔹 Check Kubernetes Version

Kubernetes supports only sequential upgrades.

✅ Allowed: v1.31 → v1.32

❌ Not Allowed: v1.31 → v1.33

Check version using:

eksctl get cluster --name <cluster-name> --region <region>
  • Cluster name and status
  • Kubernetes version
  • Region and availability zones

👉 You can also check via AWS Console → EKS → Cluster Details

🔹 Review Documentation

Always check release notes before upgrading:

  • ✨ New features
  • ⚠️ Deprecated APIs
  • ❌ Removed features
  • 🔐 Security fixes
  • ⚡ Performance improvements

🔹 Check Running Add-ons

List all add-ons before upgrade: 

kubectl get deployments -n kube-system
  • CoreDNS - DNS inside cluster
  • VPC CNI - Networking
  • Metrics Server - Metrics
  • HPA - Auto scaling
  • Istio - Service mesh
  • KEDA - Event scaling
  • Karpenter - Node provisioning

👉 Upgrade add-ons before control plane to avoid failures.

2️⃣ Upgrade Add-ons First

Before upgrading the Kubernetes control plane, upgrade all cluster add-ons.

🔹 Why Upgrade Add-ons First?

Add-ons depend on Kubernetes APIs. If not upgraded, they may break in newer versions.

  • Networking (VPC CNI)
  • DNS (CoreDNS)
  • Service Routing (kube-proxy)
  • Monitoring & Autoscaling

🔹 Upgrade Managed Add-ons (eksctl)

eksctl update addon --name coredns --cluster <cluster-name> --region <region> --force
eksctl update addon --name kube-proxy --cluster <cluster-name> --region <region> --force
eksctl update addon --name vpc-cni --cluster <cluster-name> --region <region> --force
  • CoreDNS – DNS resolution
  • kube-proxy – Networking
  • VPC CNI – Pod networking

🔹 Manual Add-on Upgrade

  • Istio
  • Prometheus
  • KEDA
kubectl apply -f https://github.com/aws/amazon-vpc-cni-k8s/releases/download/v1.11.7/amazon-vpc-cni-k8s-1.11.7.yaml

3️⃣ Upgrade the EKS Control Plane

After upgrading add-ons, upgrade the Kubernetes control plane to the target version.

🔹 Control Plane Components

  • API Server
  • Controller Manager
  • Scheduler
  • etcd

👉 Managed automatically in Amazon EKS

🔹 Upgrade using eksctl

eksctl upgrade cluster --name <cluster-name> --region <region>

👉 No downtime (non-disruptive upgrade)

🔹 Upgrade via AWS Console

  • Open AWS Console
  • Go to EKS → Clusters
  • Select cluster
  • Click "Update Version"
  • Select version

👉 Upgrade one version at a time

4️⃣ Upgrade Worker Nodes

After upgrading the control plane, upgrade worker nodes to match the Kubernetes version.

🔹 Create New Node Group

eksctl create nodegroup --cluster <cluster-name> \
--name <new-nodegroup-name> \
--node-type t3.medium \
--nodes 3 --nodes-min 1 --nodes-max 5 \
--node-ami-family AmazonLinux2 \
--version <target-version> \
--region <region>

👉 Create new node group instead of in-place upgrade

🔹 Cordon and Drain Old Nodes

kubectl cordon <node-name>

kubectl drain <node-name> \
--ignore-daemonsets \
--delete-emptydir-data

👉 Safely move workloads to new nodes

🔹 Delete Old Node Group

eksctl delete nodegroup --cluster <cluster-name> \
--name <old-nodegroup-name>

🔹 Self-Managed Nodes

  • Update Launch Template
  • Update Auto Scaling Group
  • Drain old nodes

5️⃣ Post-Upgrade Validation

After the upgrade, verify that all cluster components and workloads are running properly.

🔹 Verification Commands

kubectl get nodes
kubectl get pods --all-namespaces
kubectl get deployment -n kube-system

👉 All nodes should be Ready and pods should be Running

🎯 Interview Question

Q: Explain EKS upgrade process

  • Pre-Upgrade: Check version, review docs, list add-ons
  • Add-ons: Upgrade CoreDNS, kube-proxy, VPC CNI
  • Control Plane: Upgrade using eksctl or console
  • Nodes: Create new node group, drain old nodes
  • Validation: Verify nodes and pods
  • Rollback: Recreate nodes, downgrade add-ons

👉 Control plane rollback is NOT supported in EKS
