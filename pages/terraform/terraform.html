🔹 Terraform Basics — Deep Dive

📌 What is Infrastructure as Code (IaC)?

Infrastructure as Code (IaC) is the management of infrastructure (networks, VMs, load balancers) using declarative configuration files rather than manual processes or interactive tools. Terraform by HashiCorp is the leading IaC tool.

✅ Key benefits: Version control, repeatability, auditability, and self-documenting infrastructure. Instead of clicking through AWS console, you write .tf files and treat infrastructure like application code.

📘 Why IaC is critical in DevOps: Enables continuous delivery, eliminates configuration drift, allows teams to test infrastructure changes via pull requests, and scales with automation.

🔌 Providers (AWS, Azure, local, etc.)

A provider is a Terraform plugin that translates API calls to different cloud providers or services. Each provider offers resources like aws_instance, azurerm_virtual_machine, or google_compute_instance.

# Example: multiple providers
provider "aws" {
  region = "us-east-1"
}

provider "azurerm" {
  features {}
}

provider "local" {
  # local provider for file generation
}

Best practice: Always pin provider versions using required_providers block to ensure deterministic builds.

🛠️ Install Terraform (Cross-platform)

macOS (Homebrew): brew tap hashicorp/tap && brew install hashicorp/tap/terraform
Linux (apt): wget -O- https://apt.releases.hashicorp.com/gpg | gpg --dearmor | sudo tee /usr/share/keyrings/hashicorp-archive-keyring.gpg
Windows (Chocolatey): choco install terraform
Verify: terraform -version → should display latest version.

⚙️ Basic Commands (deep explanation)

  • terraform init → Initializes working directory, downloads provider plugins, sets up backend. Must be run first in any new config.
  • terraform plan → Creates an execution plan, showing which resources will be added, changed, or destroyed. No actual changes happen.
  • terraform apply → Applies the changes required to reach desired state. Prompts for confirmation unless -auto-approve used.
  • terraform destroy → Removes all resources defined in the configuration. Irreversible — use cautiously.
  • terraform fmt → Formats configuration files to canonical style.
  • terraform validate → Validates syntax and internal consistency.
🔥 Standard Workflow: init → fmt → validate → plan → apply → (optional: destroy)

🔹 Core Concepts — Deep Dive

🧱 Resources – Building blocks

Resources are declared using resource <type> <name> {}. Each resource maps to a physical or logical component. Terraform uses resource arguments and attributes (like id, arn) to manage lifecycle.

resource "aws_instance" "web_server" {
  ami           = "ami-0c55b159cbfafe1f0"
  instance_type = "t3.micro"
  tags = {
    Name = "Production-Web"
    Env  = "prod"
  }
}

After apply, you can reference attributes elsewhere: aws_instance.web_server.public_ip.

📥 Variables (variable.tf) – Parameterization

Variables make modules reusable. Define variable "instance_type" {} and assign via terraform.tfvars or CLI -var.

# variables.tf
variable "region" {
  description = "AWS region"
  type        = string
  default     = "us-west-2"
}

variable "environment" {
  type    = string
  sensitive = false
}

# terraform.tfvars
environment = "staging"

Order of precedence: environment variables (TF_VAR_name) > terraform.tfvars > auto.tfvars > command line -var.

📤 Outputs – Exposing values

Use output blocks to display info after apply, or to pass data between modules. Outputs are stored in state and can be queried via terraform output.

output "instance_public_ip" {
  description = "Public IP of EC2"
  value       = aws_instance.web_server.public_ip
}

output "instance_id" {
  value = aws_instance.web_server.id
}

🗂️ State File (terraform.tfstate)

The state file is a JSON mapping between your config and real resources. It tracks metadata, dependencies, and attributes. Never edit manually. For teams, use remote backends (S3, GCS, Azure Storage). State locking prevents concurrent modifications.

⚠️ Critical: If state is lost, Terraform loses track of resources → you must import them manually (terraform import).

🔗 Dependency Handling

Terraform automatically builds a dependency graph from references like aws_subnet.main.id. Implicit dependencies guarantee correct order (subnet before instance). Use depends_on for hidden dependencies or side effects.

resource "aws_s3_bucket" "example" {
  bucket = "my-app-bucket"
}

resource "aws_s3_bucket_object" "file" {
  bucket = aws_s3_bucket.example.id
  key    = "data.txt"
  source = "file.txt"
  # depends_on is implicit because bucket id is referenced
}

🔹 Intermediate Terraform – Production Patterns

📦 Modules – Reusable Infrastructure

Modules are containers for multiple resources used together. A root module calls child modules. Create a modules/ folder with standard structure: main.tf, variables.tf, outputs.tf, README.md.

# modules/ec2-cluster/main.tf
resource "aws_instance" "this" {
  count         = var.instance_count
  ami           = var.ami_id
  instance_type = var.instance_type
}

# root/main.tf
module "web_cluster" {
  source   = "./modules/ec2-cluster"
  instance_count = 3
  ami_id   = "ami-12345"
  instance_type = "t3.small"
}

Registry: Use public modules from Terraform Registry (VPC, EKS, etc.) to accelerate development.

☁️ Remote Backend (S3, Azure, GCS)

Remote backends enable team collaboration and state consistency. Best practice: S3 + DynamoDB for state locking to prevent concurrent runs.

terraform {
  backend "s3" {
    bucket         = "my-company-terraform-state"
    key            = "prod/app/terraform.tfstate"
    region         = "us-east-1"
    dynamodb_table = "terraform-locks"
    encrypt        = true
  }
}

Other backends: azurerm, gcs, consul, http. Always enable encryption.

🧩 Workspaces – Environment Isolation

Workspaces allow multiple named state instances within a single configuration. Use for dev / staging / prod but not recommended for large teams (folder structure preferred).

terraform workspace new dev
terraform workspace new prod
terraform workspace select dev

resource "aws_instance" "app" {
  instance_type = terraform.workspace == "prod" ? "c5.large" : "t3.micro"
}
💡 Alternative: Many prefer directory-based separation (envs/dev/, envs/prod/) for clearer state isolation.

🔍 Data Sources – Query existing infrastructure

Data sources fetch information from providers without creating new resources. Ideal for importing existing VPCs, AMIs, or security groups.

data "aws_ami" "ubuntu" {
  most_recent = true
  owners      = ["099720109477"] # Canonical
  filter {
    name   = "name"
    values = ["ubuntu/images/hvm-ssd/ubuntu-focal-20.04-amd64-server-*"]
  }
}

resource "aws_instance" "server" {
  ami = data.aws_ami.ubuntu.id
  instance_type = "t2.micro"
}

🔹 Advanced Terraform – Enterprise-Grade Techniques

⚙️ Provisioners (local-exec, remote-exec)

Provisioners run scripts on local or remote machines after resource creation. However, they are not recommended for primary configuration — use config management (Ansible, Chef) or user_data instead.

resource "aws_instance" "web" {
  ami           = "ami-123"
  instance_type = "t2.micro"
  user_data = <<-EOF
    #!/bin/bash
    apt update && apt install nginx -y
  EOF

  provisioner "local-exec" {
    command = "echo ${self.public_ip} >> inventory.txt"
  }

  provisioner "remote-exec" {
    inline = [
      "sudo systemctl start nginx"
    ]
    connection {
      type = "ssh"
      user = "ubuntu"
      private_key = file("~/.ssh/key.pem")
    }
  }
}
⚠️ Avoid overusing provisioners: They are not idempotent by default and can fail unpredictably. Prefer immutable images (Packer) or cloud-init.

🔁 Dynamic Blocks – Generate nested config

Dynamic blocks reduce repetition when creating multiple nested blocks (e.g., security group rules, EBS block devices).

variable "ingress_rules" {
  default = [
    { port = 80, desc = "HTTP" },
    { port = 443, desc = "HTTPS" },
    { port = 22, desc = "SSH" }
  ]
}

resource "aws_security_group" "dynamic_sg" {
  name = "dynamic-sg"
  dynamic "ingress" {
    for_each = var.ingress_rules
    content {
      from_port   = ingress.value.port
      to_port     = ingress.value.port
      protocol    = "tcp"
      description = ingress.value.desc
      cidr_blocks = ["0.0.0.0/0"]
    }
  }
}

🧠 Conditional Expressions & Meta-arguments

Ternary condition: condition ? true_val : false_val. Control resource creation with count or for_each.

resource "aws_instance" "bastion" {
  count = var.create_bastion ? 1 : 0
  ami   = data.aws_ami.linux.id
  instance_type = var.env == "prod" ? "c5.large" : "t3.micro"
}

# for_each with map
resource "aws_s3_bucket" "buckets" {
  for_each = {
    dev  = "my-dev-data"
    prod = "my-prod-data"
  }
  bucket = each.value
  tags = { Environment = each.key }
}

🌍 Multi-Environment Setup (dev/stage/prod)

Best practices for managing multiple environments:

  • Directory structure: Separate folders per environment using symlinked modules.
  • Terragrunt: Thin wrapper for DRY configurations.
  • Workspaces (simple but limited).
# Production folder: prod/main.tf
terraform {
  backend "s3" { bucket = "prod-state-bucket", key = "prod/terraform.tfstate" }
}

module "app" {
  source = "../modules/app"
  env    = "prod"
  instance_count = 5
}

# apply with var-file
terraform apply -var-file="prod.tfvars"
🎯 Pro tip: Use CI/CD pipelines (GitHub Actions, GitLab) to auto-apply changes after PR merges into environment branches.

🔹 Terraform Hands-on Projects (From Zero to Production)

🖥️ Project 1: Launch EC2 Instance

Goal: Single web server with security group, key pair, and user_data script.
Concepts: Providers, resources, variables, outputs.

provider "aws" { region = "us-east-1" }

resource "aws_security_group" "web_sg" {
  name = "web-sg"
  ingress {
    from_port   = 80
    to_port     = 80
    protocol    = "tcp"
    cidr_blocks = ["0.0.0.0/0"]
  }
  ingress {
    from_port   = 22
    to_port     = 22
    protocol    = "tcp"
    cidr_blocks = ["${chomp(data.http.myip.body)}/32"]
  }
}

data "http" "myip" { url = "https://checkip.amazonaws.com" }

resource "aws_instance" "web" {
  ami           = "ami-0c55b159cbfafe1f0"
  instance_type = "t2.micro"
  vpc_security_group_ids = [aws_security_group.web_sg.id]
  user_data = <<EOF
#!/bin/bash
echo "<h1>Hello from Terraform</h1>" > /var/www/html/index.html
systemctl start nginx
EOF
  tags = { Name = "Terraform-Demo" }
}

🌐 Project 2: Create VPC + Subnet + Internet Gateway

Goal: Custom VPC with public subnet, route table, and internet gateway to host public-facing services.

resource "aws_vpc" "main" {
  cidr_block = "10.0.0.0/16"
  enable_dns_support = true
}

resource "aws_subnet" "public" {
  vpc_id            = aws_vpc.main.id
  cidr_block        = "10.0.1.0/24"
  map_public_ip_on_launch = true
  availability_zone = "us-east-1a"
}

resource "aws_internet_gateway" "igw" { vpc_id = aws_vpc.main.id }

resource "aws_route_table" "public_rt" {
  vpc_id = aws_vpc.main.id
  route {
    cidr_block = "0.0.0.0/0"
    gateway_id = aws_internet_gateway.igw.id
  }
}

resource "aws_route_table_association" "public_assoc" {
  subnet_id      = aws_subnet.public.id
  route_table_id = aws_route_table.public_rt.id
}

🏢 Project 4: Multi-Tier Architecture (Frontend + Backend + RDS)

Goal: Production-like setup with load balancer, auto-scaling group (frontend), application servers, and RDS database. Demonstrates high availability.

# Simplified structure
resource "aws_lb" "app_lb" {
  name = "multi-tier-lb"
  internal = false
  load_balancer_type = "application"
  subnets = module.vpc.public_subnets
}

resource "aws_launch_template" "web" {
  name = "web-template"
  image_id = data.aws_ami.amazon_linux.id
  instance_type = "t3.micro"
  user_data = base64encode(file("frontend.sh"))
}

resource "aws_autoscaling_group" "frontend_asg" {
  desired_capacity = 2
  max_size = 5
  min_size = 2
  vpc_zone_identifier = module.vpc.private_subnets
  launch_template { id = aws_launch_template.web.id }
}

resource "aws_db_instance" "database" {
  engine = "mysql"
  instance_class = "db.t3.micro"
  allocated_storage = 20
  db_name = "myappdb"
  username = "admin"
  password = random_password.db_password.result
  vpc_security_group_ids = [aws_security_group.db_sg.id]
  skip_final_snapshot = true
}

module "vpc" {
  source  = "terraform-aws-modules/vpc/aws"
  version = "5.0.0"
  name = "multi-tier-vpc"
  cidr = "10.0.0.0/16"
  private_subnets = ["10.0.1.0/24", "10.0.2.0/24"]
  public_subnets  = ["10.0.101.0/24", "10.0.102.0/24"]
  enable_nat_gateway = true
}
🚀 Learning outcome: Understand real-world architecture patterns, load balancing, autoscaling, stateful services (RDS), and network segmentation (public/private subnets).
⚠️ Cost awareness: Multi-tier projects may incur costs. Always run terraform destroy after learning.
