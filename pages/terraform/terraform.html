🚀 Terraform Complete Guide

🔹 Terraform Basics

📌 What is Infrastructure as Code (IaC)?

Infrastructure as Code (IaC) is the practice of managing and provisioning infrastructure through machine-readable configuration files instead of manual processes.

Instead of logging into cloud consoles and creating resources manually, you define everything in code (.tf files), and Terraform automatically provisions it.

Why IaC is important:
  • ✔ Automation (no manual work)
  • ✔ Consistency (same infra every time)
  • ✔ Version control (track changes like code)
  • ✔ Faster deployments

🔌 Providers

Providers are plugins that allow Terraform to interact with external platforms such as AWS, Azure, GCP, or even local systems.

Each provider exposes a set of resources and APIs that Terraform can manage. 

provider "aws" {
  region = "us-east-1"
}
Key Point: Without a provider, Terraform cannot create or manage any infrastructure.

⚙️ Basic Terraform Commands

These are the core commands used in every Terraform workflow:

  • terraform init → Initializes the working directory and downloads providers
  • terraform plan → Shows execution plan (what will be created/changed)
  • terraform apply → Applies the changes and creates infrastructure
  • terraform destroy → Deletes all managed infrastructure
Typical Workflow:
init → plan → apply → (modify) → plan → apply → destroy

🔹 Core Concepts

🧱 Resources

Resources are the core building blocks in Terraform. They represent real infrastructure such as EC2 instances, S3 buckets, VPCs, etc.

Each resource has a type, name, and configuration. 

resource "aws_instance" "web" {
  ami           = "ami-123"
  instance_type = "t2.micro"
}

Terraform compares this code with the current state and decides what to create, update, or delete.

📥 Variables

Variables make your Terraform code reusable and flexible. Instead of hardcoding values, you define variables that can change based on environment. 

variable "region" {
  default = "us-east-1"
}

You can override variables using:

  • CLI arguments
  • tfvars files
  • Environment variables

📤 Outputs

Outputs display useful information after Terraform runs. They are commonly used to show resource details like IP address, DNS name, or IDs. 

output "public_ip" {
  value = aws_instance.web.public_ip
}

Outputs can also be used to pass data between modules.

🗂️ State File (terraform.tfstate)

The state file stores the current state of your infrastructure. Terraform uses it to map your configuration to real resources.

  • Keeps track of existing resources
  • Helps Terraform detect changes
  • Used during updates and deletions

Without the state file, Terraform cannot understand what already exists.

⚠️ Do not edit the state file manually. In real projects, store it remotely (S3, etc.).

🔗 Dependencies

Terraform automatically manages the order of resource creation based on dependencies.

Example: 

resource "aws_instance" "web" {
  subnet_id = aws_subnet.my_subnet.id
}

Here, Terraform understands that the subnet must be created before the EC2 instance.

You can also define dependencies manually: 

depends_on = [aws_vpc.main]

🔹 Intermediate Terraform

📦 Modules

Modules are reusable Terraform configurations. Instead of writing the same code again and again, you can group resources into a module and reuse it across projects.

Example structure: 

modules/
  vpc/
    main.tf
    variables.tf
    outputs.tf

Using a module: 

module "vpc" {
  source = "./modules/vpc"
  cidr_block = "10.0.0.0/16"
}
Why use modules?
  • ✔ Reusability
  • ✔ Cleaner code
  • ✔ Easy maintenance
  • ✔ Standardized infrastructure

☁️ Remote Backend

By default, Terraform stores state locally. In real projects, this is unsafe and not suitable for teams.

A remote backend stores the state file in a shared location like AWS S3. 

terraform {
  backend "s3" {
    bucket = "my-terraform-state"
    key    = "dev/terraform.tfstate"
    region = "us-east-1"
  }
}
Benefits:
  • ✔ Team collaboration
  • ✔ Prevents state loss
  • ✔ Centralized management
⚠️ Always use remote backend in production environments.

🧩 Workspaces

Workspaces allow you to manage multiple environments (dev, staging, production) using the same Terraform code.

Each workspace has its own state file. 

terraform workspace new dev
terraform workspace new prod
terraform workspace select dev

Example use: 

instance_type = terraform.workspace == "prod" ? "t2.large" : "t2.micro"
Use Case: Run same code but different configurations for different environments.

🔍 Data Sources

Data sources allow you to fetch existing resources instead of creating new ones.

Example: Use an existing VPC instead of creating one. 

data "aws_vpc" "existing" {
  default = true
}

You can then use it like: 

vpc_id = data.aws_vpc.existing.id
Key Difference:
  • resource → creates infrastructure
  • data → reads existing infrastructure

🔹 Advanced Terraform

⚙️ Provisioners

Provisioners are used to execute scripts or commands on a resource after it is created. They are typically used for bootstrapping (installing software, configuring servers). 

resource "aws_instance" "web" {
  ami           = "ami-123"
  instance_type = "t2.micro"

  provisioner "local-exec" {
    command = "echo Instance created!"
  }
}

Types of provisioners:

  • local-exec → runs on your local machine
  • remote-exec → runs on the created resource
⚠️ Provisioners should be used as a last resort. Prefer configuration tools like Ansible, Docker, or cloud-init.

🔁 Dynamic Blocks

Dynamic blocks allow you to generate repeated nested blocks programmatically. Useful when you need multiple similar configurations. 

dynamic "ingress" {
  for_each = var.ports
  content {
    from_port   = ingress.value
    to_port     = ingress.value
    protocol    = "tcp"
  }
}

This avoids writing repetitive code manually.

Use Case: Creating multiple security group rules dynamically.

🧠 Conditional Expressions

Terraform supports conditional logic using ternary expressions. 

instance_type = var.env == "prod" ? "t2.large" : "t2.micro"

You can also control resource creation: 

count = var.env == "prod" ? 1 : 0
Use Case: Create expensive resources only in production.

🌍 Multi-Environment Setup

In real projects, you manage multiple environments like:

  • Development (dev)
  • Staging (test)
  • Production (prod)

This can be done using:

  • Workspaces
  • Separate variable files (dev.tfvars, prod.tfvars)
  • Separate folders/projects
terraform apply -var-file="dev.tfvars"
terraform apply -var-file="prod.tfvars"

Example configuration difference: 

instance_type = var.env == "prod" ? "t2.large" : "t2.micro"
Best Practice: Use same code, different configurations for each environment.

🔹 Terraform Projects

🖥️ Launch EC2 Instance

This is the simplest Terraform project. It creates a basic EC2 instance. 

provider "aws" {
  region = "us-east-1"
}

resource "aws_instance" "web" {
  ami           = "ami-123"
  instance_type = "t2.micro"

  tags = {
    Name = "MyEC2"
  }
}

🏗️ Full Infrastructure (EC2 + S3 + Security Group)

This project combines multiple services into a working infrastructure. 

resource "aws_security_group" "web_sg" {
  name = "web-sg"

  ingress {
    from_port   = 80
    to_port     = 80
    protocol    = "tcp"
    cidr_blocks = ["0.0.0.0/0"]

resource "aws_s3_bucket" "bucket" {
  bucket = "my-terraform-bucket"
}

resource "aws_instance" "web" {
  ami           = "ami-123"
  instance_type = "t2.micro"
  security_groups = [aws_security_group.web_sg.name]
}
What you learn:
  • Multiple resource integration
  • Security configuration
  • Real-world infrastructure setup
